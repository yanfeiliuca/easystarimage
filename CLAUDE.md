# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

EasyStarImage (易星写真) — a single-page app where a user uploads 1-3 face photos, picks a photography-style "template" (e.g. 张曼玉 red-carpet gown, 高圆圆 art-minimalist), and gets back an AI-generated portrait with their face composited into that style via Replicate's InstantID model.

Runtime: a single Cloudflare Worker, no build step, no framework, no package.json.

## Critical architecture note: `api/generate.js` is the only thing deployed

`wrangler.toml` sets `main = "api/generate.js"`. That one file contains:
- The entire HTTP router (`handleRequest`)
- The full template registry (`const TEMPLATES = {...}`, ~1000 lines)
- The **entire frontend** inlined as template-literal strings: `const HTML`, `const CSS`, `const JS`, served verbatim at `/`, `/css/style.css`, `/js/app.js`

The loose files at the repo root — `index.html`, `css/style.css`, `js/app.js`, `templates.js` — are **not served by the Worker and are not referenced by wrangler**. They exist only as human-readable copies for editing convenience, and `templates.js` documents itself as `// Auto-generated by scripts/build_template_registry.py` (that script is not present in this repo).

**These copies routinely drift from what's actually embedded in `api/generate.js` and deployed.** As of this writing:
- `css/style.css` matches the embedded `CSS` const exactly.
- `index.html` matches the embedded `HTML` const exactly.
- `js/app.js` does **not** match the embedded `JS` const (the embedded version has newer error-handling code, e.g. `showError(msg)` vs. the file's `showError()`).
- `templates.js` does **not** match the embedded `TEMPLATES` (e.g. Zhang Manyu `category` fields were unified to `` `张曼玉` `` only inside `generate.js`).

**Implication for any change:** editing `index.html` / `css/style.css` / `js/app.js` / `templates.js` alone has zero effect on the live site. To actually ship a frontend or template change, edit the corresponding `const HTML` / `const CSS` / `const JS` / `const TEMPLATES` block inside `api/generate.js` (and, ideally, mirror the same edit into the standalone file so it doesn't drift further — but the embedded copy is what matters).

## Commands

```bash
# Local dev server (serves the Worker exactly as deployed, from api/generate.js)
npx wrangler dev

# Deploy
npx wrangler deploy

# Set the Replicate API token (required — the Worker 500s on every request without it)
npx wrangler secret put REPLICATE_API_TOKEN
```

There is no lint/test/build tooling in this repo.

⚠️ `wrangler.toml`'s comment says to run `wrangler secret put EVOLINK_API_KEY` — that's stale/wrong. The code reads `env.REPLICATE_API_TOKEN` (see `handleRequest` in `api/generate.js`). `EVOLINK_API_KEY` is a leftover from an earlier API provider and is not used anywhere.

## Request flow

1. `GET /api/templates` — frontend loads template metadata (id/name/category/description/tags/shotCount only; prompt text is withheld from the client).
2. User uploads photo(s) client-side, picks a template + shot number.
3. `POST /api/generate` `{ template, shot, images: [dataURI, ...] }` — server builds the prompt via `buildPrompt(templateId, shot)` / `buildNegativePrompt(templateId)`, takes `images[0]` as the InstantID face reference (multiple uploaded photos are accepted by the UI but **only the first is ever sent to the model**), and calls `submitReplicateTask` → returns `{ task_id }` (a Replicate prediction ID).
4. Frontend polls `GET /api/status/:taskId` until `status: 'completed'` (or `'failed'`).
5. `GET /api/proxy?url=...` streams the final Replicate-hosted image back through the Worker (adds `Content-Disposition: attachment` and, when the URL is on `replicate.delivery`/`replicate.com`, the auth header) so the browser can download it without a cross-origin fetch.
6. `GET /api/debug/:taskId` is a leftover raw-response passthrough to Replicate, explicitly commented "调试用，部署后可删除" (debug-only, meant to be deleted post-deploy).

The user's photo is never persisted server-side — it's forwarded to Replicate and discarded (the `TEMP_IMAGES` KV namespace declared in `wrangler.toml` is provisioned but currently unused by any code path).

## Template registry shape

Each entry in `TEMPLATES` (keyed by template id):
```js
{
  id, name, styleName, category, description,
  format: "raw" | "zm_v1" | "zm_v2",   // metadata only — not read by any code, purely descriptive of which prompt-writing convention was used
  tags: [...],
  shots: { "1": "prompt text...", "2": "...", ... },  // per-shot/composition prompt variants, 1-8 shots per template
  commonBase: "",                       // optional string appended to every shot's prompt
  negativeConstraints: "...",          // optional, prepended to the shared negative prompt in buildNegativePrompt
}
```
`buildPrompt` concatenates `shots[shot]` + `commonBase` + a fixed quality suffix ("Ultra-high definition, photorealistic, 8K quality..."). `buildNegativePrompt` concatenates `negativeConstraints` + a fixed base negative prompt.

`shot` defaults to `1` and falls back to shot `"1"` if the requested shot number doesn't exist. `template` defaults to `zhang_man_yu_17` if unspecified.

## Frontend structure (`js/app.js`, mirrored in the `JS` const)

Single self-invoking function, no modules, plain DOM APIs (`var`, not `let`/`const`, to match the rest of the file's style — no build/transpile step, this ships as-is to the browser). Key pieces:
- Three upload "slots" supporting click, drag-drop, and clipboard-paste, each holding one `File`.
- Template `<select>` populated from `/api/templates`; selecting a template shows its info card and populates a shot `<select>`.
- `startGeneration` → POSTs to `/api/generate`, then `pollTask` loops against `/api/status/:id` with a timeout, then routes to `showResult` or `showError`.
- `currentTaskId` / `isGenerating` guard against duplicate submissions and let "重试" (retry) resume polling an in-flight task instead of resubmitting.
